What is Redis?

Remote Dictionary Server — blazing-fast in-memory data store

In-Memory

All data lives in RAM. Reads/writes are microsecond-fast — 100k+ ops/sec is typical.

Persistent

Optional RDB snapshots + AOF logs. Survive restarts without losing data.

Multi-Structure

Strings, Hashes, Lists, Sets, Sorted Sets, Streams, Bitmaps, HyperLogLog.

When to Use Redis

⚡

Application Caching

Cache DB queries, API responses, computed results. Reduce latency by 10–100x.

🔐

Session Management

Store user sessions with TTL. Works across multiple app instances.

🛡️

Rate Limiting

Block brute force attacks. Throttle API calls per user or IP.

📡

Pub / Sub Messaging

Real-time notifications, live feeds, microservice communication.

🏆

Leaderboards

Connection Pool Setup

RedisConfig.java 
import redis.clients.jedis.*;

public class RedisConfig {

    public static JedisPool createPool() {
        JedisPoolConfig config = new JedisPoolConfig();
        config.setMaxTotal(20);          // max connections
        config.setMaxIdle(10);           // keep 10 idle
        config.setMinIdle(2);            // always keep 2 ready
        config.setTestOnBorrow(true);   // validate before use

        return new JedisPool(
            config,
            "localhost",   // host
            6379,          // port
            2000,          // connection timeout ms
            "yourPassword" // password
        );
    }

    // Usage pattern — always use try-with-resources!
    public static void example(JedisPool pool) {
        try (Jedis jedis = pool.getResource()) {
            jedis.set("key", "value");
            String val = jedis.get("key");
        } // connection returned to pool automatically
    }
}
💡
Pool tip: Always use JedisPool (not single Jedis) in any multi-threaded app. Jedis is NOT thread-safe. Pool handles concurrency for you.

Jedis Client Basics

Core commands you'll use every day

BasicRedisOperations.java 
try (Jedis jedis = pool.getResource()) {

    // ── SET / GET ──────────────────────────────────────────
    jedis.set("name", "Alice");
    String name = jedis.get("name");         // "Alice"

    // SET with TTL (expire in 60 seconds)
    jedis.setex("token", 60, "abc123");
    long ttl = jedis.ttl("token");            // 60 (seconds left)

    // ── EXISTENCE & DELETE ────────────────────────────────
    boolean exists = jedis.exists("name");    // true
    jedis.del("name");                         // delete key
    jedis.expire("token", 120);               // update TTL

    // ── COUNTERS ───────────────────────────────────────────
    jedis.set("visits", "0");
    jedis.incr("visits");                     // → 1 (atomic!)
    jedis.incrBy("visits", 5);               // → 6
    jedis.decr("visits");                     // → 5

    // ── MULTI-GET (pipeline-friendly) ─────────────────────
    jedis.mset("k1", "v1", "k2", "v2");
    List<String> vals = jedis.mget("k1", "k2");

    // ── SET ONLY IF NOT EXISTS (lock pattern) ────────────
    long set = jedis.setnx("lock:resource", "owner1");
    // 1 = acquired, 0 = already locked

    // ── PIPELINE (batch commands, one round-trip) ─────────
    Pipeline pipe = jedis.pipelined();
    pipe.set("a", "1");
    pipe.set("b", "2");
    pipe.set("c", "3");
    pipe.sync();   // sends all 3 in one network call
}

Strings & Key Design

The fundamental data type + naming patterns

Key Naming Convention

📌
Use colon-separated namespaces: entity:id:field — e.g. user:42:profile, product:101:price, session:abc123
StringOperations.java 
public class StringRedisService {

    // Store JSON as string (common pattern)
    public void cacheUserJson(Jedis jedis, int userId, String json) {
        String key = "user:" + userId + ":profile";
        jedis.setex(key, 3600, json); // expires in 1 hour
    }

    // Atomic counter (page views, likes, etc.)
    public long incrementPageViews(Jedis jedis, String page) {
        String key = "views:" + page;
        long count = jedis.incr(key);
        // Set expiry only on first increment (new key)
        if (count == 1) jedis.expire(key, 86400); // 24h
        return count;
    }

    // Distributed lock (simple version)
    public boolean acquireLock(Jedis jedis, String resource, String owner) {
        SetParams params = SetParams.setParams()
            .nx()           // NX = only set if Not eXists
            .ex(30);        // EX = expire in 30 seconds
        String result = jedis.set("lock:" + resource, owner, params);
        return "OK".equals(result);
    }

    public void releaseLock(Jedis jedis, String resource, String owner) {
        String key = "lock:" + resource;
        // Only delete if WE own it (Lua script = atomic)
        String script = "if redis.call('get',KEYS[1])==ARGV[1] " +
                         "then return redis.call('del',KEYS[1]) else return 0 end";
        jedis.eval(script, 1, key, owner);
    }
}

Hashes

Like a Java Map stored under one key. Perfect for objects.

💡
Use Hash when storing an object with multiple fields. Much more memory efficient than one String key per field.
HashOperations.java 
public class UserService {

    // Store a user object as Hash
    public void saveUser(Jedis jedis, User user) {
        String key = "user:" + user.getId();

        // HMSET — set multiple fields at once
        Map<String, String> fields = new HashMap<>();
        fields.put("name", user.getName());
        fields.put("email", user.getEmail());
        fields.put("role", user.getRole());
        fields.put("createdAt", String.valueOf(user.getCreatedAt()));
        jedis.hset(key, fields);
        jedis.expire(key, 7200); // 2 hours
    }

    // Get all fields of a user
    public Map<String, String> getUser(Jedis jedis, int userId) {
        return jedis.hgetAll("user:" + userId);
    }

    // Get only specific fields (more efficient)
    public List<String> getUserBasic(Jedis jedis, int userId) {
        return jedis.hmget("user:" + userId, "name", "email");
    }

    // Update a single field without loading the whole object
    public void updateUserRole(Jedis jedis, int userId, String role) {
        jedis.hset("user:" + userId, "role", role);
    }

    // Increment a numeric field (e.g., login count)
    public long incrementLoginCount(Jedis jedis, int userId) {
        return jedis.hincrBy("user:" + userId, "loginCount", 1);
    }

    // Check field exists
    public boolean hasEmail(Jedis jedis, int userId) {
        return jedis.hexists("user:" + userId, "email");
    }
}

Lists

Doubly-linked lists. Great for queues, feeds, and recent items.

ListOperations.java 
public class TaskQueueService {

    // ── QUEUE PATTERN (FIFO) ───────────────────────────────
    // Producer: push to left
    public void enqueue(Jedis jedis, String task) {
        jedis.lpush("queue:emails", task);
    }

    // Consumer: pop from right (FIFO order)
    public String dequeue(Jedis jedis) {
        return jedis.rpop("queue:emails");
    }

    // Blocking pop — waits up to 5s for a task
    public String blockingDequeue(Jedis jedis) {
        List<String> result = jedis.brpop(5, "queue:emails");
        // result = [queueName, value] or null on timeout
        return result != null ? result.get(1) : null;
    }

    // ── RECENT ITEMS (Activity Feed) ───────────────────────
    public void addActivity(Jedis jedis, int userId, String activity) {
        String key = "feed:" + userId;
        jedis.lpush(key, activity);           // newest first
        jedis.ltrim(key, 0, 99);              // keep only 100 items
    }

    public List<String> getRecentActivity(Jedis jedis, int userId) {
        return jedis.lrange("feed:" + userId, 0, 19); // first 20
    }

    // Queue size
    public long queueSize(Jedis jedis) {
        return jedis.llen("queue:emails");
    }
}

Caching Layer

Cache-aside pattern — the most common Redis use case

📋
Cache-Aside Pattern: Check Redis first → if miss, load from DB → store in Redis → return. On update, invalidate the cache key.
ProductCacheService.java 
@Service
public class ProductCacheService {

    private final JedisPool jedisPool;
    private final ProductRepository db;
    private final ObjectMapper mapper = new ObjectMapper();

    private static final int TTL_SECONDS = 3600; // 1 hour

    // ── CACHE-ASIDE GET ────────────────────────────────────
    public Product getProduct(long productId) throws Exception {
        String key = "product:" + productId;

        try (Jedis jedis = jedisPool.getResource()) {
            // 1. Check cache
            String cached = jedis.get(key);
            if (cached != null) {
                // Cache HIT ✓
                return mapper.readValue(cached, Product.class);
            }

            // 2. Cache MISS — load from database
            Product product = db.findById(productId)
                .orElseThrow(() -> new RuntimeException("Not found"));

            // 3. Store in cache with TTL
            String json = mapper.writeValueAsString(product);
            jedis.setex(key, TTL_SECONDS, json);

            return product;
        }
    }

    // ── CACHE INVALIDATION ────────────────────────────────
    public Product updateProduct(Product product) throws Exception {
        // Update DB first
        Product updated = db.save(product);

        // Then invalidate cache (next read will populate fresh)
        try (Jedis jedis = jedisPool.getResource()) {
            jedis.del("product:" + product.getId());
        }
        return updated;
    }

    // ── BULK CACHE (warmup) ───────────────────────────────
    public void warmUpCache(List<Product> products) throws Exception {
        try (Jedis jedis = jedisPool.getResource()) {
            Pipeline pipe = jedis.pipelined();
            for (Product p : products) {
                String json = mapper.writeValueAsString(p);
                pipe.setex("product:" + p.getId(), TTL_SECONDS, json);
            }
            pipe.sync(); // all stored in one round-trip!
        }
    }
}
⚠️
Cache Stampede: If many requests hit an expired key simultaneously, they all go to DB. Fix with a short mutex lock on the first refill, or use Redis WAIT.

Session Store

Stateless app servers + Redis = horizontal scaling

SessionService.java 
public class SessionService {

    private static final int SESSION_TTL = 1800; // 30 minutes

    // Create session after login
    public String createSession(Jedis jedis, User user) {
        String sessionId = UUID.randomUUID().toString();
        String key = "session:" + sessionId;

        // Store session data as Hash
        Map<String, String> sessionData = new HashMap<>();
        sessionData.put("userId", String.valueOf(user.getId()));
        sessionData.put("email", user.getEmail());
        sessionData.put("role", user.getRole());
        sessionData.put("createdAt", Instant.now().toString());

        jedis.hset(key, sessionData);
        jedis.expire(key, SESSION_TTL);

        return sessionId; // send as cookie to client
    }

    // Validate and get session (also refreshes TTL)
    public Map<String, String> getSession(Jedis jedis, String sessionId) {
        String key = "session:" + sessionId;
        Map<String, String> session = jedis.hgetAll(key);

        if (session.isEmpty()) return null; // expired or invalid

        // Sliding expiry — reset TTL on each request
        jedis.expire(key, SESSION_TTL);
        return session;
    }

    // Logout
    public void destroySession(Jedis jedis, String sessionId) {
        jedis.del("session:" + sessionId);
    }

    // Track all sessions for a user (to invalidate all on logout)
    public void trackUserSession(Jedis jedis, int userId, String sessionId) {
        jedis.sadd("user_sessions:" + userId, sessionId);
    }

    public void logoutAllDevices(Jedis jedis, int userId) {
        Set<String> sessions = jedis.smembers("user_sessions:" + userId);
        if (!sessions.isEmpty()) {
            String[] keys = sessions.stream()
                .map(s -> "session:" + s)
                .toArray(String[]::new);
            jedis.del(keys);
            jedis.del("user_sessions:" + userId);
        }
    }
}

Rate Limiting

Protect APIs from abuse. Two strategies: Fixed Window & Sliding Window.

RateLimiter.java 
public class RateLimiter {

    // ── STRATEGY 1: Fixed Window Counter ──────────────────
    // 100 requests per user per minute
    public boolean allowFixed(Jedis jedis, String userId) {
        // Key resets every minute
        long window = System.currentTimeMillis() / 60000;
        String key = "rate:" + userId + ":" + window;

        long count = jedis.incr(key);
        if (count == 1) jedis.expire(key, 70); // 70s safety margin
        return count <= 100; // true = allowed
    }

    // ── STRATEGY 2: Sliding Window (more accurate) ────────
    // Uses Sorted Set with timestamps as scores
    public boolean allowSliding(Jedis jedis, String userId,
                                 int maxRequests, int windowSeconds) {
        String key = "sliding:" + userId;
        long now = System.currentTimeMillis();
        long windowStart = now - (windowSeconds * 1000L);

        Pipeline pipe = jedis.pipelined();

        // Remove expired entries outside the window
        pipe.zremrangeByScore(key, 0, windowStart);

        // Count requests in current window
        Response<Long> count = pipe.zcard(key);

        // Add this request
        pipe.zadd(key, now, now + "-" + Math.random());
        pipe.expire(key, windowSeconds + 1);
        pipe.sync();

        return count.get() < maxRequests;
    }

    // ── STRATEGY 3: Token Bucket (smooth bursting) ────────
    public boolean allowTokenBucket(Jedis jedis, String userId) {
        // Use Lua script for atomic refill + consume
        String lua = 
            "local tokens = tonumber(redis.call('get', KEYS[1]) or ARGV[1]) " +
            "local refill = tonumber(ARGV[2]) " +
            "local max = tonumber(ARGV[1]) " +
            "local now = tonumber(ARGV[3]) " +
            "local last = tonumber(redis.call('get', KEYS[2]) or now) " +
            "local elapsed = (now - last) / 1000 " +
            "tokens = math.min(max, tokens + elapsed * refill) " +
            "if tokens >= 1 then " +
            "  redis.call('set', KEYS[1], tokens - 1) " +
            "  redis.call('set', KEYS[2], now) " +
            "  return 1 " +
            "end " +
            "return 0";

        long result = (Long) jedis.eval(lua, 2,
            "bucket:" + userId,    // KEYS[1] = token count
            "bucket_ts:" + userId,  // KEYS[2] = last refill time
            "100",                   // ARGV[1] = max tokens
            "10",                    // ARGV[2] = refill rate (tokens/sec)
            String.valueOf(System.currentTimeMillis())
        );
        return result == 1;
    }
}

Real-Time Leaderboard

Sorted Sets are perfect for rankings. O(log N) for all operations.

LeaderboardService.java 
@Service
public class LeaderboardService {

    private static final String BOARD = "leaderboard:global";

    // Add or update a player's score
    public void addScore(Jedis jedis, String playerId, double score) {
        jedis.zadd(BOARD, score, playerId);
        // Also store player profile in hash
        jedis.hset("player:" + playerId, "score", String.valueOf(score));
    }

    // Increment (e.g., kill in a game = +10 points)
    public double incrementScore(Jedis jedis, String playerId, double points) {
        return jedis.zincrby(BOARD, points, playerId);
    }

    // Get top N with scores and ranks
    public List<LeaderboardEntry> getTopPlayers(Jedis jedis, int n) {
        List<LeaderboardEntry> entries = new ArrayList<>();
        List<Tuple> results = new ArrayList<>(
            jedis.zrevrangeWithScores(BOARD, 0, n - 1)
        );

        for (int i = 0; i < results.size(); i++) {
            Tuple t = results.get(i);
            entries.add(new LeaderboardEntry(
                i + 1,           // rank
                t.getElement(),  // playerId
                t.getScore()    // score
            ));
        }
        return entries;
    }

    // Get surrounding players (my rank ± 2)
    public List<LeaderboardEntry> getMyContext(Jedis jedis, String playerId) {
        Long rank = jedis.zrevrank(BOARD, playerId);
        if (rank == null) return Collections.emptyList();

        long from = Math.max(0, rank - 2);
        long to = rank + 2;

        return new ArrayList<>(
            jedis.zrevrangeWithScores(BOARD, from, to)
        ).stream().map(/* map to LeaderboardEntry */...).collect(...);
    }

    // Get player's exact rank
    public long getMyRank(Jedis jedis, String playerId) {
        Long rank = jedis.zrevrank(BOARD, playerId);
        return rank != null ? rank + 1 : -1;
    }
}

@Service
public class RedisTemplateService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    public void saveObject(String key, Object value, long ttlSeconds) {
        redisTemplate.opsForValue().set(key, value,
            Duration.ofSeconds(ttlSeconds));

        return redisTemplate.opsForValue().get(key);
    }

    }

    public void addToZSet(String key, Object value, double score) {
        redisTemplate.opsForZSet().add(key, value, score);
    }
}

Best Practices

Production-ready patterns and things to avoid

DO ✓
  • Always use try-with-resources for Jedis
  • Use JedisPool (thread-safe), not Jedis directly
  • Namespace keys: entity:id:field
  • Always set TTL to prevent memory leaks
  • Use Pipeline for bulk operations
  • Use Lua scripts for atomic multi-step ops
  • Monitor with redis-cli INFO stats
AVOID ✗
  • Never use KEYS * in production (blocks server)
  • Don't store huge objects (keep values <100KB)
  • Avoid long-running Lua scripts
  • Don't treat Redis as primary DB for important data
  • Avoid reusing Jedis across threads
  • Don't store sensitive data without encryption
  • Don't ignore connection errors / timeouts

// ── EXPIRY CONSTANTS ─────────────────────────────────────
int ONE_MINUTE  = 60;
int ONE_HOUR    = 3600;
int ONE_DAY     = 86400;
int ONE_WEEK    = 604800;

// ── WHICH DATA STRUCTURE TO USE? ─────────────────────────
// String  → single value, counter, JSON blob, lock
// Hash    → object with multiple fields (user, product)
// List    → queue, stack, activity feed, recent items
// Set     → unique tags, online users, mutual friends
// ZSet    → leaderboard, priority queue, sliding rate limit

// ── DEBUG COMMANDS (redis-cli) ────────────────────────────
// MONITOR             – stream all commands (dev only!)
// INFO memory         – memory usage stats
// DBSIZE              – total key count
// SCAN 0 MATCH user:* – safely iterate matching keys
